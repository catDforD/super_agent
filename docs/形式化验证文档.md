# 形式化验证输入与自动转换说明

> **版本**: v2.1
> **更新日期**: 2026-06-29
> **状态**: 已实现并稳定运行

## 1. 目标

本文档说明当前项目中「形式化验证」的最新输入规范、支持的输入方式、自动转换规则，以及如何确保输入最终落到 PLCverif 可执行的标准格式。

当前主要执行链路：

- `LangChain/api/multi_agent_api.py` - API 入口
- `LangChain/capabilities/verification.py` - 形式化验证模块 (`FormalVerificationMixin`)
- `LangChain/validators/syntax.py` - ST 语法验证 (`STSyntaxValidator`, `FormalValidationInputParser`)
- `LangChain/validators/property_fixer.py` - 属性修复 (`PropertyFixer`)
- `src/plcverif.py` - PLCverif 工具封装
- `src/routes/formal_routes.py` - 形式化验证 API 路由

## 2. 统一标准格式（最终落地格式）

无论你用哪种输入方式，后端都会尽量归一化为下面这种结构后再调用 PLCverif：

```json
[
  {
    "property_description": "中文描述 / English description",
    "property": {
      "job_req": "pattern",
      "pattern_id": "pattern-invariant",
      "pattern_params": {
        "1": "instance.var >= 0"
      },
      "entry_point": "FB_Name"
    }
  }
]
```

说明：

- `job_req`：`assertion` 或 `pattern`
- `pattern` 模式必须有 `pattern_id` 和 `pattern_params`
- `assertion` 模式：断言不会从 `pattern_params` 推导，而是依赖你在 `st_code` 里写入的断言注释（形如 `//#ASSERT <左值> = <右值> : <label>`）
- `FUNCTION_BLOCK/PROGRAM` 变量建议使用 `instance.xxx`
- 多 POU 场景建议提供 `entry_point`

## 3. 当前支持的输入方式

> 重要说明（避免歧义）：自然语言是**输入源**，系统会把自然语言规则转换为形式化验证属性（`properties` 标准结构）；不是"把内容转换成自然语言"。

### 3.1 方式 A：标准 JSON（推荐）

```json
{
  "st_code": "FUNCTION_BLOCK Example\nVAR_INPUT\n    x : BOOL;\nEND_VAR\nVAR_OUTPUT\n    y : BOOL;\nEND_VAR\ny := x;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "输出 y 必须等于输入 x / y must equal x",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": { "1": "instance.y = instance.x" },
        "entry_point": "Example"
      }
    }
  ]
}
```

### 3.2 方式 B：简写 JSON（兼容，自动转换）

```json
{
  "st_code": "FUNCTION_BLOCK Example\nVAR_INPUT\n    x : BOOL;\nEND_VAR\nVAR_OUTPUT\n    y : BOOL;\nEND_VAR\ny := x;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property": "safety",
      "description": "输出 y 必须等于输入 x / y must equal x"
    }
  ]
}
```

该写法会在验证前自动转换为标准 `property` 对象（`pattern-invariant`）。

### 3.3 方式 C：描述型对象（兼容，自动转换）

```json
{
  "st_code": "...",
  "properties": [
    {
      "description": "count 始终非负"
    }
  ]
}
```

后端会自动补齐 `job_req/pattern_id/pattern_params`。

### 3.4 方式 D：`properties` 传纯文本（兼容，自动转换）

```json
{
  "st_code": "...",
  "properties": "count 始终非负；alarm 触发时必须为 TRUE"
}
```

后端会按分行/分号拆分为多条属性并自动构造 `pattern-invariant` 模板。

### 3.5 方式 E：只给 ST 代码（自动生成属性）

```json
{
  "st_code": "FUNCTION_BLOCK ... END_FUNCTION_BLOCK"
}
```

如果未提供属性，工作流会进入 `property` 阶段自动生成属性，再进行验证：

- 位置：`LangChain/capabilities/verification.py::_execute_property`
- 触发条件：当工作流状态 `state.properties` 为空时触发自动生成。
- 生成逻辑：读取 `last_st_file_path` 中的 ST 代码，并结合 `state.input`（用户需求）调用 property agent 生成标准 `properties` JSON。
- 兜底策略：如果没有可用的 property agent，则使用默认断言属性（`job_req: "assertion"`）确保流程不中断。
- 后续处理：进入 `verification` 前，系统还会调用 `PropertyFixer.fix_properties(...)` 对 properties 做必要的格式/字段修复，降低因结构问题导致的伪失败。

### 3.6 方式 F：自然语言直输（自动转换为形式化验证输入）

```json
{
  "st_code": "FUNCTION_BLOCK ... END_FUNCTION_BLOCK",
  "natural_language_properties": "1) 计数器始终非负；2) requestIn 为 TRUE 时，count 最终应大于 0"
}
```

说明：

- 可直接提交，后端会自动转换为标准 `properties` 结构后再验证。
- 前端按钮"转为验证属性 JSON（可选）"仅用于提前预览/人工微调转换结果，不是必点步骤。

### 3.7 方式 G：使用 `property_requirements` 字段（兼容自然语言）

```json
{
  "st_code": "FUNCTION_BLOCK ... END_FUNCTION_BLOCK",
  "property_requirements": "当 requestIn 为 TRUE 时，输出 y 必须在同周期为 TRUE"
}
```

说明：

- `property_requirements` 与 `natural_language_properties` 语义等价，都会被自动转换为标准 `properties`。

### 3.8 方式 H：JSON 属性 + 自然语言补充（混合输入）

```json
{
  "st_code": "FUNCTION_BLOCK ... END_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "输出 y 必须等于输入 x",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "instance.y = instance.x"
        }
      }
    }
  ],
  "natural_language_properties": "另外补充：报警状态与安全状态不能同时为 FALSE"
}
```

说明：

- 系统会优先使用你提供的 JSON 属性，并将自然语言补充转换后合并。

### 3.9 方式 I：单属性对象（非数组，兼容）

```json
{
  "st_code": "FUNCTION_BLOCK ... END_FUNCTION_BLOCK",
  "properties": {
    "property_description": "验证断言",
    "property": {
      "job_req": "assertion"
    }
  }
}
```

说明：

- 后端会自动包装成数组格式后再验证。

## 4. 自动转换规则（关键）

自动转换主要由 `PropertyFixer` 执行：

- 文件：`LangChain/validators/property_fixer.py`
- 主要调用点：形式化验证失败后的反例调试阶段  
  （`LangChain/capabilities/verification.py::_execute_debugging_counterexample` 中的自动修复步骤）

核心规则：

1. **简写属性转标准结构**  
   `{"property":"safety","description":"..."}` → `pattern-invariant`

2. **从描述提取表达式（启发式）**  
   优先识别比较表达式（如 `x >= 0`），其次识别常见语义（如"非负""不能同时为 TRUE 和 FALSE"）。

3. **变量前缀规范化**  
   `FUNCTION_BLOCK/PROGRAM` 场景下尽量自动补 `instance.` 前缀，并修复 `FB_Name.var -> instance.var`。

4. **失败兜底**  
   语义不清晰时会生成可验证占位表达式，保证流程不中断；建议你在提交前手工确认 `pattern_params` 是否符合真实需求。

## 5. 工作流行为（与迭代相关）

## 5.1 验证前先规范化属性

在进入 PLCverif 之前，系统会先执行属性自动修复/转换，再调用验证器，减少因为属性格式问题导致的伪失败。

## 5.2 反例调试后的路由优化

已修复逻辑：

- 当 `debugging_counterexample` 已成功修复代码或属性时，下一步直接回到 `verification`
- 不再无条件先跳 `coding`，避免"属性问题却反复改代码"的无效迭代

对应路由：`LangChain/core/routers.py::SmartRouter._route_for_validation_mode`

## 5.3 未成功检查（not checked）场景的处理

已增加失败类型区分：

- 若存在 `violated`，按"代码/属性违例"处理，可进入反例调试。
- 若仅有 `not successfully checked`（尤其是 `always true` 恒真属性），会判定为"属性可验证性问题"，不再反复改 ST 代码。

说明：

- `always true` 通常表示属性写得过弱或语义恒真，不是代码 bug。
- 这类情况建议优先修改 `pattern_params` 或属性模式，再重跑验证。

## 6. ST语言属性验证示例文档

### 6.1 概述

本文档整理了ST（Structured Text）语言中属性验证的典型示例，涵盖**不变式（invariant）**、**蕴含（implication）**、**断言（assertion）** 三种核心验证模式，结合FUNCTION_BLOCK（FB）、FUNCTION等不同POU（Program Organization Unit）类型，说明属性验证的写法、语义匹配要求及通用注意事项，帮助理解ST代码属性验证的核心逻辑和规范。

### 6.2 核心验证模式示例

#### 示例1：单个FB不变式（最易验证通过）

**业务含义**：每个周期将y赋值为x，因此周期末恒有x与y相等。

**ST代码**：
```st
FUNCTION_BLOCK FB_Twin
VAR
    x : BOOL := FALSE;
    y : BOOL := FALSE;
END_VAR
y := x;
END_FUNCTION_BLOCK
```

**属性JSON（pattern-invariant，预期为真）**：
```json
{
  "property_description": "周期末 y 恒等于 x",
  "property": {
    "job_req": "pattern",
    "pattern_id": "pattern-invariant",
    "pattern_params": {
      "1": "instance.x = instance.y"
    }
  }
}
```

**说明**：
- FB类型POU默认使用`instance.变量名`引用变量；若需显式指定POU名，可添加`"entry_point": "FB_Twin"`。
- 属性中布尔值写法与ST保持一致（TRUE/FALSE），可提升验证稳定性。

#### 示例2：带上下界的计数器不变式

**业务含义**：计数器c初始值为0，仅当inc为真且c<100时自增，需验证c始终非负。

**ST代码**：
```st
FUNCTION_BLOCK FB_Counter
VAR
    c : INT := 0;
    inc : BOOL := FALSE;
END_VAR
IF inc THEN
    IF c < 100 THEN
        c := c + 1;
    END_IF;
END_IF;
END_FUNCTION_BLOCK
```

**属性JSON（pattern-invariant，预期为真）**：
```json
{
  "property_description": "c 不会变成负数",
  "property": {
    "job_req": "pattern",
    "pattern_id": "pattern-invariant",
    "pattern_params": {
      "1": "instance.c >= 0"
    }
  }
}
```

**说明**：不变式模式用于验证"某条件在所有周期始终成立"，此示例聚焦数值范围约束，是工业场景中计数器安全验证的典型写法。

#### 示例3：蕴含模式（安全响应验证）

**业务含义**：当请求req为真且状态ok为假（非法请求）时，需置位错误标识err；验证逻辑为"若req且非ok，则err必为真"。

**ST代码**：
```st
FUNCTION_BLOCK FB_Guard
VAR
    req : BOOL := FALSE;
    ok  : BOOL := TRUE;
    err : BOOL := FALSE;
END_VAR
IF req AND NOT ok THEN
    err := TRUE;
END_IF;
END_FUNCTION_BLOCK
```

**属性JSON（pattern-implication，预期为真）**：
```json
{
  "property_description": "非法请求时 eventually 反映为 err（按周期末语义）",
  "property": {
    "job_req": "pattern",
    "pattern_id": "pattern-implication",
    "pattern_params": {
      "0": "instance.req = TRUE AND instance.ok = FALSE",
      "1": "instance.err = TRUE"
    },
    "entry_point": "FB_Guard"
  }
}
```

**说明**：
- 蕴含模式核心是"前提成立则结论必成立"，前提为假时整体属性自动成立，适合验证"触发条件→响应动作"的安全逻辑。
- 需确保周期末语义匹配：若代码中非法请求触发后本周期未置位err，可能出现反例，需保证代码与属性语义一致。

#### 示例4：FUNCTION蕴含模式（非负输入验证）

**业务含义**：函数F_Add实现两整数相加，需验证"当输入a、b均非负时，返回值也非负"。

**ST代码**：
```st
FUNCTION F_Add : INT
VAR_INPUT
    a : INT;
    b : INT;
END_VAR
F_Add := a + b;
END_FUNCTION
```

**属性JSON（pattern-implication，预期为真）**：
```json
{
  "property_description": "非负输入时返回值非负",
  "property": {
    "job_req": "pattern",
    "pattern_id": "pattern-implication",
    "pattern_params": {
      "1": "F_Add.a >= 0 AND F_Add.b >= 0",
      "2": "F_Add.ret_val >= 0"
    }
  }
}
```

**说明**：
- FUNCTION类型POU的变量引用规则：`函数名.输入变量`，返回值固定用`ret_val`。
- 无需`instance.`前缀，与FB/PROGRAM区分开。

#### 示例5：断言模式（ST内写断言）

**业务含义**：直接在ST代码中嵌入断言语句，验证变量x始终非负。

**ST代码**：
```st
FUNCTION_BLOCK FB_Assert
VAR 
    x : INT := 0; 
END_VAR
//#ASSERT (x >= 0);
x := x;
END_FUNCTION_BLOCK
```

**属性JSON（assertion模式）**：
```json
{
  "property_description": "检查 //#ASSERT",
  "property": {
    "job_req": "assertion"
  }
}
```

**说明**：
- 断言模式极简：属性JSON仅指定`job_req: "assertion"`，断言逻辑直接写在ST中（以`//#ASSERT`标识）。
- 验证工具会扫描ST中的断言语句并检查是否成立。

### 6.3 通用使用要点

| 要点 | 详细说明 |
|------|----------|
| 变量引用限定词 | 常规约定：FB/PROGRAM用`instance.变量名`；FUNCTION用`函数名.变量名`，返回值用`ret_val` |
| 多POU场景 | 必须在属性JSON中添加`"entry_point": "目标POU名"`，明确验证的POU |
| 表达式写法 | 减法需写为`a = 0 - b`；属性中避免出现除零、未定义运算（如负数开方） |
| 验证"通过"判定 | 需满足两个条件：工具链执行成功 + 属性与代码语义完全一致；超时/unknown结果不代表数学上的"证真" |

### 6.4 当前版本兼容说明（重要）

在当前仓库集成的 `plcverif-cli (1.0.0.202410210930)` 环境中，`job_req=pattern` 对部分布尔等式表达式会出现解析兼容问题（典型报错：`pattern-based requirement representation ... SyntaxError ... mismatched input`）。

常见表现：

- `pattern_params.1 = "instance.y = instance.x"` 报语法错误；
- 回退到 cbmc 后最终结果为 `not_checked_reason=no_suitable_files`；
- 这不等同于代码逻辑错误，更多是 pattern 语法解析阶段失败。

**建议策略（稳定优先）**：

1. **布尔等式类属性优先使用 `assertion`**（在 ST 中写 `//#ASSERT`）。
2. 只有在 pattern 表达式已知可解析时再使用 `job_req=pattern`。
3. 遇到连续 `not_checked + no_suitable_files`，优先改属性表达方式，不要反复改 ST 业务逻辑。

## 7. 推荐输入模板（可直接用）

> 实战建议：如果目标是"先稳定验证通过"，优先使用 **7.2 assertion 模板**。
> 尤其是"输出等于输入 / 布尔等式"这类场景，assertion 比 pattern 更稳。

### 7.1 标准模板

```json
{
  "st_code": "FUNCTION_BLOCK Example\nVAR_INPUT\n    x : BOOL;\nEND_VAR\nVAR_OUTPUT\n    y : BOOL;\nEND_VAR\ny := x;\n//#ASSERT (y = x) : assert_y_equals_x\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "输出 y 必须等于输入 x / y must equal x",
      "property": {
        "job_req": "assertion"
      }
    }
  ]
}
```

如果是单 POU，`entry_point` 可省略；多 POU 建议显式提供。

### 7.2 多 POU 模板（建议显式 `entry_point`）

```json
{
  "st_code": "FUNCTION_BLOCK FB_A\nVAR_OUTPUT\n    y : BOOL;\nEND_VAR\ny := TRUE;\nEND_FUNCTION_BLOCK\n\nFUNCTION_BLOCK FB_B\nVAR_OUTPUT\n    z : BOOL;\nEND_VAR\nz := FALSE;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "验证 FB_A 的输出 y 为 TRUE",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "instance.out_y"
        },
        "entry_point": "FB_A"
      }
    }
  ]
}
```

### 7.3 `assertion` 模板

```json
{
  "st_code": "FUNCTION_BLOCK FB_Assert\nVAR_INPUT\n    in1 : BOOL;\nEND_VAR\nVAR_OUTPUT\n    out1 : BOOL;\nEND_VAR\nout1 := in1;\n//#ASSERT out1 = in1 : assert_out1_matches_in1\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "验证代码中的断言",
      "property": {
        "job_req": "assertion"
      }
    }
  ]
}
```

输入注意事项：
- `job_req=assertion` 时，真正的断言需要写在 ST 里，形如：`//#ASSERT <左值> = <右值> : <label>`
- `//#ASSERT` 是注释行，不要写成普通 ST 语句（例如不要用 `;` 结尾的伪语句）
- `in/out` 变量名请与 `//#ASSERT` 表达式中的变量一致

### 7.4 `pattern-implication` 模板（传感器到报警）

```json
{
  "st_code": "FUNCTION_BLOCK FB_Imp\nVAR_INPUT\n    sensor : BOOL;\nEND_VAR\nVAR_OUTPUT\n    alarm : BOOL;\nEND_VAR\nalarm := sensor;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "如果 sensor 为 TRUE，那么同一 PLC 周期末 alarm 必为 TRUE",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-implication",
        "pattern_params": {
          "1": "instance.sensor",
          "2": "instance.alarm"
        },
        "entry_point": "FB_Imp"
      }
    }
  ]
}
```

输入注意事项：
- `pattern-implication` 需要两个参数：`pattern_params.1`（触发条件）+ `pattern_params.2`（期望条件）
- 变量必须使用 `instance.` 前缀（`FUNCTION_BLOCK/PROGRAM` 场景）
- 布尔常量用 `TRUE/FALSE`

### 7.5 `pattern-forbidden` 模板（禁止错误状态）

```json
{
  "st_code": "FUNCTION_BLOCK FB_Forbidden\nVAR_OUTPUT\n    error : BOOL;\nEND_VAR\nerror := FALSE;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "error 在 PLC 周期末不可能为 TRUE（禁止错误状态）",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-forbidden",
        "pattern_params": {
          "1": "instance.error"
        },
        "entry_point": "FB_Forbidden"
      }
    }
  ]
}
```

输入注意事项：
- `pattern-forbidden` 只需要一个参数，通常填 key `1`
- 你的表达式应是"错误状态的判定"，例如：`instance.error = TRUE`

### 7.6 `pattern-invariant` 模板（不变式 / 范围）

```json
{
  "st_code": "FUNCTION_BLOCK FB_Range\nVAR_OUTPUT\n    count : INT;\nEND_VAR\ncount := 0;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "count 在 PLC 周期末恒非负（不变式）",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "instance.count >= 0"
        },
        "entry_point": "FB_Range"
      }
    }
  ]
}
```

输入注意事项：
- `pattern-invariant` 只需要一个参数（`pattern_params.1`）
- 表达式尽量写成可比较的范围/关系式，例如 `>= / <= / =`

### 7.7 多 POU 多属性模板（为每条属性指定 `entry_point`）

```json
{
  "st_code": "FUNCTION_BLOCK FB_A\nVAR_OUTPUT\n    y : BOOL;\nEND_VAR\ny := TRUE;\nEND_FUNCTION_BLOCK\n\nFUNCTION_BLOCK FB_B\nVAR_OUTPUT\n    z : BOOL;\nEND_VAR\nz := FALSE;\nEND_FUNCTION_BLOCK",
  "properties": [
    {
      "property_description": "验证 FB_A 的输出 y 必为 TRUE",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "instance.out_y"
        },
        "entry_point": "FB_A"
      }
    },
    {
      "property_description": "验证 FB_B 的输出 z 必为 FALSE",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "!instance.out_z"
        },
        "entry_point": "FB_B"
      }
    }
  ]
}
```

输入注意事项：
- `st_code` 可以包含多个 `FUNCTION_BLOCK`，但建议每条 property 都显式提供 `entry_point`
- `entry_point` 必须与 ST 里实际的 `FUNCTION_BLOCK` 名称一致

### 8.2 请求入口与模式选择

- 前端 / API 将形式化验证输入（通常是 JSON 字符串）传给后端工作流入口
  - 入口函数：`LangChain/api/multi_agent_api.py::multi_agent_workflow_stream`
  - 模式：`workflow_mode="formal_validation"`
- 该函数会：
  - 识别是否为「形式化验证模式」
  - 调用 `FormalValidationInputParser` 对输入做结构化解析
  - 把解析结果（ST 代码 + properties）传入工作流管理器

### 8.3 输入解析与基础校验（FormalValidationInputParser）

- 位置：`LangChain/validators/syntax.py::FormalValidationInputParser`
- 主要职责：
  1. 从原始输入中解析出：
     - `st_code`：ST 源代码
     - `properties`：验证属性（可以是标准 JSON / 简写 / 自然语言等多种形态）
  2. 提取规则补充：
     - 若输入本身是 JSON，则直接读取 `properties/st_code` 字段，并对 `properties` 的 list/string 形态做标准化。
     - 若输入不是标准 JSON（纯文本），会在文本中尝试提取形如 `[...]` 的 properties JSON 数组，并做基础校验。
  2. 根据本文件第 2～3 章约定，对输入做结构化校验：
     - 必要字段是否存在（如 `st_code`、`properties`）
     - JSON 是否能正确解析
     - 明显不合法的模式（例如 mode 与字段组合冲突）
  3. 返回统一结构：
     - `has_st_code / has_properties` 标志
     - `st_code / properties` 标准化结果
     - `validation.is_valid / errors / warnings` 基础校验信息

若 `validation.is_valid = False`，工作流在「输入阶段」直接返回错误，不会进入 PLCverif 调用。

### 8.4 状态初始化与 ST 文件落地

- 位置：`LangChain/core/workflow.py::AgentExecutor._initialize_state`
- 关键步骤：
  1. 创建本次工作流的结果目录 `result/workflow_时间戳_哈希`
  2. 如果已经解析出了 `st_code`：
     - 先通过 `STSyntaxValidator.validate_and_fix` 做一轮语法/结构修复
     - 把修复后的 ST 代码落地为 `.st` 文件
     - 在工作流状态中记录：
       - `last_st_file_path`：当前待验证的 ST 文件路径
       - `st_files_path`：所有候选 ST 文件列表（用于保留最近 N 个版本）
  3. 把属性字符串（JSON / 其他形式）保存在状态的 `properties` 字段中

> 注意：此时尚未调用 PLCverif。在进入 `_execute_verification`、调用 PLCverif **之前**，会对当前 ST 与 `properties` 做一次 **`PropertyFixer.fix_properties`（及 pattern 布尔等式等预处理）**，以减少属性格式问题导致的伪失败。

### 8.5 进入形式化验证阶段（verification）

- 工作流在 `formal_validation` 模式下，会根据路由器 `SmartRouter` 的决策进入 `verification` 智能体：
  - 实现位置：`LangChain/capabilities/verification.py::_execute_verification`

在该阶段：

1. **确认前置条件**：
   - `last_st_file_path` 存在且文件存在
   - `properties` 不为空

2. **通过 `PLCverifTool.verify_st_code` 调用底层 PLCverif 工具**：
   - 参数：
     - `st_file_path`：当前 ST 文件路径
     - `properties`：属性字符串（后端内部会再解析）
     - `base_dir`：本次工作流的结果目录
   - 返回：
     - `satisfied: bool`：整体是否满足
     - `result: dict`：详细验证结果（包括每条属性的 `satisfied/violated/not_checked`，以及摘要说明）

3. **把结果反馈到工作流状态**：
   - `status.last_st_file_verified = True`
   - `status.last_st_file_validation_satisfied = satisfied`
   - 同时把带有完整 JSON 结果的 `ToolMessage("验证结果：...")` 追加到 `messages`，供后续反例调试智能体使用

### 8.6 验证结果的细粒度分类

虽然 `_execute_verification` 只把 `satisfied` 作为整体结果，但后续逻辑会进一步细分：

- 若 `satisfied = True`：
  - 所有属性满足，直接判定为「验证通过」
- 若 `satisfied = False`：
  - 在验证结果中进一步区分：
    - 是否存在 `violated = True` 的属性（**真实违例，有代码/属性反例**）
    - 是否仅存在 `not_checked = True`，且伴随 `always true` 等提示（**属性可验证性问题 / 恒真属性**）
- 在路由器中（`LangChain/core/routers.py::_route_for_validation_mode`），会依据这些信息决定：
  - 进入 `debugging_counterexample`（有真实违例时）
  - 仅提示「未成功检查/恒真属性」，不再反复修改 ST 代码

### 8.7 验证结束与工作流收尾

- 工作流结束时由 `_create_end_message` 决定最终文案：
  - `formal_validation` 模式下：
    - 若 `status.last_st_file_validation_satisfied = True` → 输出「✅ 形式化验证成功」
    - 否则 → 输出「⚠️ 形式化验证未通过」，并留给上层 UI/调用方结合详细结果做进一步判断

## 9. 反例调试机制（debugging_counterexample）

当形式化验证阶段判定「存在违例」时，路由器会进入 `debugging_counterexample` 智能体，对代码/属性进行自动修复尝试。

### 9.1 第 1 步：自动修复（不调用 LLM）

位置：`LangChain/capabilities/verification.py::_execute_debugging_counterexample` 的前半部分。

触发条件：

- 当前 `status.fixing_times == 0`（第一次进入反例调试）
- 且已经有：
  - 当前 ST 代码（`last_st_file_path` 对应的文件内容）
  - 当前属性字符串 `properties`

自动修复过程：

1. **ST 代码自动修复**
   - 调用 `STSyntaxValidator.validate_and_fix(current_code)`
   - 典型修复内容：
     - 补全 `END_VAR` / `END_IF` 等结构
     - 修正 `=` → `:=`、`&&` → `AND` 等运算符错误
     - 补分号、补类型等语法层面的修复
   - 若修复后代码与原代码不同，则视为 `code_changed = True`

2. **属性自动修复（PropertyFixer）**
   - 若 `state.properties` 不为空：
     - 调用 `PropertyFixer.fix_properties(state["properties"], current_code)`
     - 典型修复内容：
       - 自动补齐缺失字段（如 `pattern_id`、`pattern_params`）
       - 变量前缀归一化为 `instance.xxx`
       - 修正常见 JSON / 键类型问题
   - 若返回的 `fixes_applied` 非空：
     - 更新 `state["properties"]` 为修复后的 JSON 字符串
     - 输出「属性自动修复：N 处」提示

3. **决定是否立即返回重新验证**
   - 如果 `code_changed` 或 `props_changed`：
     - 对 ST 代码调用 `code_processor.validate_and_save(...)`，生成新的 `.st` 文件
     - `status.fixing_times += 1`
     - 输出「将重新验证...」提示后直接 `return`
     - 路由器在下一轮会重新进入 `verification` 智能体，基于修复后的代码/属性再跑一轮 PLCverif

> 这一阶段可以理解为「规则+启发式」的自动修复，适合处理**结构/格式类错误**，例如变量前缀、缺字段、简单语法问题等。

### 9.2 第 2 步：LLM 反例调试（基于验证结果）

当满足以下任一条件时，会进入真正的 LLM 反例调试阶段：

- 第 1 次自动修复没有对代码/属性做出任何实质修改
- 或者已经做过至少 1 次修复（`fixing_times > 0`），仍然存在违例

此时流程如下：

1. 从 `state.messages` 中取出最近一次包含「验证结果」的 `ToolMessage`：
   - 该消息内容形如：`"验证结果：\n{...完整 JSON...}"`
   - JSON 中包含每条属性的 `satisfied/violated/not_checked` 以及 `summary` 文本

2. 读取当前 ST 代码（`last_st_file_path` 对应内容）

3. 构造 LLM 提示词：
   - 含义大致为：
     - 给出当前 ST 代码
     - 给出验证失败的 JSON 结果（包含反例 / 违例摘要）
     - 要求「根据反例分析并修复 ST 代码，输出完整修复后的代码」

4. 通过 `AgentManager.get_debugging_agent_counterexample()` 获取专门的调试智能体，并调用：
   - `AgentExecutor._stream_agent_output("debugging_counterexample", prompt, model)`
   - 将流式输出累积到 `accumulated` 字符串

5. 使用 `CodeProcessor.extract_code(accumulated, current_code)` 从 LLM 输出中抽取**完整的新 ST 代码**：
   - 优先解析 `[FixedCode] ... ```st ...``` ...` 结构
   - 其次回退到「最后一个看起来像完整 ST 程序的代码块」
   - 若都失败，再退回到通用的 ST 代码提取器

6. 如果成功抽取到 `fixed_code`，则：
   - 通过 `code_processor.validate_and_save(fixed_code, state, "st_file_fixed")` 落地为新文件
   - 调用 `_cleanup_old_st_files` 保留最近 N 个版本
   - 输出「代码修复完成」提示
   - `status.fixing_times += 1`

> 重要：当前实现中，**LLM 阶段只直接修改 ST 代码，不会修改属性 JSON**；属性层面的 LLM 调整需要额外设计专门的智能体或 Prompt。

### 9.3 与编译调试（debugging_compilation）的区别

- `debugging_compilation`：
  - 目标：修复**编译错误**（语法、类型、编译器报错等）
  - 入口：`_execute_compilation` 发现编译失败后
  - 工具链：可选的 `EnhancedCompilationDebugger` + LLM
- `debugging_counterexample`：
  - 目标：修复**形式化验证中的语义违例**（有反例时）
  - 入口：`verification` 阶段发现存在 `violated` 属性
  - 工具链：`STSyntaxValidator` + `PropertyFixer` + LLM

## 10. 常见失败原因与处理

1. **属性结构不完整**  
   缺失 `job_req/pattern_id/pattern_params`。  
   处理：使用标准格式，或使用简写后检查自动转换结果。

2. **变量命名不符合 PLCverif 约定**  
   `FB_Name.var`、裸变量等。  
   处理：统一改为 `instance.var`（FUNCTION_BLOCK/PROGRAM）。

3. **`pattern_params` 语义与代码不一致**  
   格式虽然合法，但业务上不成立，验证会给反例。  
   处理：根据反例调整 ST 逻辑或调整属性表达式。

4. **多 POU 未指定入口**  
   处理：增加 `entry_point`。

5. **属性写成恒真（always true）导致"未成功检查"**  
   例如：`NOT (instance.y = TRUE AND instance.y = FALSE)`（对 BOOL 变量天然恒真）。  
   处理：改成有约束力的业务属性，如 `instance.y = instance.x`。

### 12.2 案例 1：非负不变式违例（启发式修复成功）

**场景**：计数器允许变为负数，违反非负不变式。

#### 原始 ST 代码

```st
FUNCTION_BLOCK FB_Counter
VAR_INPUT
    Inc : BOOL;
    Dec : BOOL;
END_VAR
VAR_OUTPUT
    Count : INT;
END_VAR
IF Inc THEN
    Count := Count + 1;
END_IF;
IF Dec THEN
    Count := Count - 1;
END_IF;
END_FUNCTION_BLOCK
```

#### 验证属性

```json
{
  "properties": [
    {
      "property_description": "计数器 Count 永远不会变成负数",
      "property": {
        "job_req": "pattern",
        "pattern_id": "pattern-invariant",
        "pattern_params": {
          "1": "instance.Count >= 0"
        },
        "entry_point": "FB_Counter"
      }
    }
  ]
}
```

#### 验证结果

```json
{
  "satisfied": false,
  "property_results": [
    {
      "property_index": 1,
      "property_description": "计数器 Count 永远不会变成负数",
      "status": "violated",
      "violated": true,
      "counterexample": {
        "initial_state": {
          "Count": 0,
          "Inc": false,
          "Dec": true
        },
        "steps": [
          {
            "Count": 0,
            "Dec": true
          },
          {
            "Count": -1,
            "Dec": true
          }
        ],
        "final_state": {
          "Count": -1
        },
        "summary": "在 Dec=TRUE 时，Count 从 0 变为 -1"
      }
    }
  ],
  "summary": "1 个属性违例，0 个通过，0 个未检查"
}
```

#### 调试过程

**Stage 1 - 自动修复**：
- `STSyntaxValidator.validate_and_fix()`：语法无问题
- `PropertyFixer.fix_properties()`：属性格式正确
- 结果：Stage 1 未做任何修改

**Stage 2 - 启发式修复**：
- 检测到 `pattern-invariant` + `>= 0` 模式
